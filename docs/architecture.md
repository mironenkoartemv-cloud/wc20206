# Архитектура мини-проекта Chexx WC 2026 Predictor

## Текущее состояние

Проект сейчас работает как статический фронт:

- `index.html` отвечает за структуру экранов.
- `styles.css` отвечает за визуальный слой.
- `app.js` содержит состояние, механику турнира, сохранение прогнозов, загрузку фактических результатов и расчет очков.
- `scripts/score-simulation.mjs` служит диагностическим стендом для проверки начисления очков.

Отдельного backend сейчас нет. Все данные сохраняются в браузере пользователя через `localStorage`.

## Хранение данных

Используются три ключа:

- `wc2026-predictor-draft` — текущий черновик игрока: имя, группы, выбранные третьи места, пики сетки, текущий экран.
- `wc2026-predictor-predictions` — сохраненные прогнозы игроков в текущем браузере.
- `wc2026-actual-results` — нормализованные фактические результаты турнира из внешнего источника.

Важно: `localStorage` не подходит для настоящего конкурса среди коллег, потому что данные живут только в браузере конкретного пользователя. Для реального запуска нужен общий backend.

## Поток экранов

1. Hero: пользователь вводит имя.
2. Групповой этап: пользователь расставляет 48 команд по 12 группам.
3. Лучшие третьи места: пользователь выбирает 8 команд, которые проходят дальше с третьих мест.
4. Плей-офф: пользователь выбирает победителей матчей до финала и матча за 3-е место.
5. Подтверждение: показывается выбранный чемпион.
6. Live-таблица: показываются текущие очки игроков и сохраненный прогноз пользователя.

Экран управляется через `state.stage` и список `FLOW_STAGES`.

## Источник фактических результатов

Текущий источник:

- `https://worldcup26.ir/get/groups`
- `https://worldcup26.ir/get/games`

Функция `refreshActualResultsFromSource()` читает данные, а затем `normalizeWorldCupApiResults()` приводит их к формату приложения:

- `actual.groups` — фактический порядок команд в группах.
- `actual.thirdGroups` — группы, из которых вышли лучшие третьи места.
- `actual.picks` — победители матчей плей-офф по id матчей.

## Начисление очков

Расчет выполняет `calculateLiveScore(record, actual)`.

Текущие правила:

- Группа полностью угадана: `+3`.
- Ровно 2 точных места в группе: `+2`.
- Ровно 1 точное место в группе: `+1`.
- Команда верно выбрана среди 8 лучших третьих мест: `+2`.
- Победитель 1/16: `+1`.
- Победитель 1/8: `+2`.
- Победитель 1/4: `+3`.
- Победитель 1/2: `+4`.
- Победитель матча за 3-е место: `+5`.
- Победитель финала: `+10`.

Максимум при идеальном прогнозе:

- Группы: `12 * 3 = 36`.
- Третьи места: `8 * 2 = 16`.
- Плей-офф: `16 + 16 + 12 + 8 + 5 + 10 = 67`.
- Итого: `119`.

## Целевая backend-архитектура

Подробная схема backend описана в [`backend-architecture.md`](./backend-architecture.md).

Для настоящего конкурса стоит вынести данные из браузера:

- `POST /players` — создать или найти игрока.
- `POST /predictions` — сохранить прогноз игрока.
- `GET /predictions/me` — получить сохраненный прогноз текущего игрока.
- `GET /leaderboard` — получить live-таблицу.
- `POST /admin/actual-results/refresh` или cron job — обновить фактические результаты из источника.

Минимальные таблицы:

- `players`: `id`, `name`, `created_at`.
- `predictions`: `id`, `player_id`, `groups_json`, `third_groups_json`, `picks_json`, `champion_id`, `saved_at`.
- `actual_results`: `id`, `source`, `groups_json`, `third_groups_json`, `picks_json`, `updated_at`.
- `score_snapshots` опционально: кеш рассчитанных очков, если участников станет много.

Фронт при такой схеме перестает быть источником истины: он только отправляет прогноз, читает live-таблицу и показывает сохраненный выбор.
